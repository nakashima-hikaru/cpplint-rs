## 2026-04-15 - Extract Regex compilation to thread_local cache for string operators
**Learning:** In hot loops such as string replacement functions (`add_spaces_around_operator`), dynamically compiling regexes on every call becomes a noticeable overhead. The set of inputs (operators) to these regexes is very small, meaning caching by input string provides a large performance boost. But be careful when introducing caches to avoid global lock contention which can sometimes be slower.
**Action:** Use `thread_local!` with `std::cell::RefCell` and a `FxHashMap` cache for compiled `std::sync::Arc<Regex>` objects to gain performance without incurring multi-threaded locking overheads.

## 2024-05-23 - Extract Regex compilation to thread_local cache for multi-threaded global lock performance
**Learning:** In highly parallel multi-threaded environments, attempting to optimize by moving `Regex` compilation into a global map inside a `RwLock` or `Mutex` (e.g., wrapped in `LazyLock`) often leads to severe lock contention. The overhead of acquiring locks constantly per thread ends up negating the benefits of the cache completely or making it slower.
**Action:** When caching repeatedly compiled expressions like `Regex` where the inputs might differ significantly, implement a `thread_local!` cache using `std::cell::RefCell` mapped to the cache structure. This eliminates the locking overhead while preserving caching, resulting in higher throughput for thread-pool based tasks like those in `rayon`.
